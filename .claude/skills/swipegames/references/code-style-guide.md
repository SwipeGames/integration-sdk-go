# Go Code Style Guide

## Repository Structure
- All services organized in `services/` directory
- Common libraries also in their own directories
- `docs/` folder contains platform documentation

## Layered Architecture

Use layered architecture with clear separation:
- `controller` - HTTP/gRPC handlers, request validation
- `service` - Business logic orchestration
- `repository` - Database operations
- `usecases` - Complex business logic (if needed)

**Versioning**: Use versioning in `controller` and `usecases` packages:
- `controller/v1`, `usecases/v1`
- Allows multiple API versions support

### File Naming
- **Omit folder name in file name**: `controller/v1/user.go` NOT `controller/v1/user-controller.go`
- **Common functionality**: Name as folder - `controller/v1/controller.go`
- **Package names**: Short version of folder structure - `controllerv1` (omit special symbols)

## Model Separation

Make clean separation between three model types:

1. **API Model**: Requests/responses from OpenAPI spec
   - Package: `api`
   - Generated from OpenAPI specifications

2. **Domain Model**: Business logic entities
   - Package: `domain`
   - Used within services and usecases

3. **Persistence Model**: Database entities
   - Package: `repository`
   - Generated by go-jet from database schema

## Domain Types

Use domain types for all parameters:
- If method accepts `gameID`, create `GameID` domain type
- Makes code more readable
- Helps avoid type confusion mistakes

Example:
```go
type GameID string
type SessionID uuid.UUID

func GetGame(gameID GameID) (*Game, error)
```

## Naming Conventions

### Function/Method Names
- Don't repeat package name: `GetUser` NOT `GetUserFromControllerV1`
- Called outside as `controllerv1.GetUser`

### Import Naming
When you have similar imports (domain from common lib and domain from service):
- Current service import: NO suffix
- Common lib import: `2` suffix
- Additional imports: `3`, `4`, etc.

Example:
```go
import (
    domain "github.com/swipegames/platform/services/core/internal/domain"
    domain2 "github.com/swipegames/platform-lib-common/domain"
)
```

## Input Validation

All validation happens in controller layer or value objects before passing to business logic.

### CID Validation
Use domain types from common library (`platform-lib-common`):

```go
import domain2 "github.com/swipegames/platform-lib-common/domain"

cid, err := domain2.NewCIDFromString(req.Cid.String())
if err != nil {
    return domain2.NewBadRequestError("CID is required", nil)
}
```

### String/ExtCID Validation
Use utils from common library (`platform-lib-common`):

```go
import utils2 "github.com/swipegames/platform-lib-common/utils"

if utils2.IsEmpty(req.ExtCID) {
    return domain2.NewBadRequestError("ExtCID is required", nil)
}
```

### Validation Best Practices
- Validate all required fields in controller
- Use domain constructors (e.g., `NewCIDFromString`) for type safety
- Use utils helpers (e.g., `IsEmpty`) for string validation
- Return `domain.BadRequestError` for validation failures
- Don't pass invalid data to service layer

## Common Libraries

### platform-lib-common (Primary Common Library)
**Repository**: `platform-lib-common`
**Location**: All shared code, utilities, and domain types
**Import pattern**: Always use `2` suffix to avoid conflicts with service-local packages

**Key packages**:
- `domain` - Shared domain types (CID, domain.Error, etc.)
- `utils` - Utility functions (IsEmpty, string helpers, etc.)
- `controller` - Shared controller utilities
- `tests` - Integration test suite base

**Example imports**:
```go
import (
    domain2 "github.com/swipegames/platform-lib-common/domain"
    utils2 "github.com/swipegames/platform-lib-common/utils"
    tests2 "github.com/swipegames/platform-lib-common/tests"
)
```

### services/internal (Common Library)
- Shared functionality used across all Go services within platform
- Import with `2` suffix: `controller2 "github.com/swipegames/platform-lib-common/controller"`

### services/internal/integration (Common Integration Library)
- Used for integration adapters only
- Provide exact names: `-service` suffix for services
- All reside in same `integration` package

### services/internal/game (Common Game Library)
- Used for games only
- Provide exact names: `-service` suffix for services
- All reside in same `game` package

## Error Handling

- Use error wrapping to pass errors through layers
- Specify root error cause on lower layer
- Top layer (controller) returns errors as-is (no wrapping)
- Don't log errors when returning them (middleware logs)
- Between services use `domain.Error` (even for gRPC)

### Logging Guidelines
- **Debug**: Development flow information (omitted in production)
- **Info**: Critical operations (game/money related) for tracing
- **Never log**: Secrets, API keys, financial amounts in plain text
- **Context logging**: Use `.Ctx(ctx)` for span/trace IDs

## Comments
- Write comments starting with lowercase letter
- Example: `// this is a comment` NOT `// This is a comment`

## Code Quality
- Don't use named returns in long methods/functions
- Prefer integration tests over unit tests
- Every service needs main test suite based on `tests.IntegrationTestSuite`

## API Development

### Versioning
- Use semver: `v1.0.0`
- Minor versions always backward-compatible
- Optional fields for backward compatibility

### Standards
- All times in UTC timezone, RFC3339 format: `2024-01-30T15:04:05Z`
- Money as string in main currency unit: `10.50` for $10.50
- All request validation in controller/handler layer

### OpenAPI operationId Naming

## Testing

### Test Organization
- Prefer integration tests in `tests/` directory
- Use test suites based on base integration suite
- Group related tests in subfolders with common suite file
- Unit tests in same package as code they test

### Test Data
- **Static Testing**: Use fixtures from `db/fixtures`
- **Dynamic Testing**: Create data on the fly, clean up after test
- Use `goose` to manage fixtures

### Critical Logging
All money-changing operations must be logged:
- Log level: Info
- Include: game session ID, round ID, external requests/responses
- Use `.Ctx(ctx)` for distributed tracing
